# 네비게이션 리뉴얼 프로토타입 비교

## 현재 문제점
- HomeClient.tsx 721줄, 11개 섹션 수직 나열
- 스크롤 5000px+ → 하단 기능(캘린더, 즐겨찾기, K-pop) 발견율 극히 낮음
- 기능이 많아졌지만 네비게이션 구조가 없음

## 각 프로토타입 상세

### A. Bottom Tab Navigation (`A_BottomTabNav.tsx`)
```
[🍽️ 추천] [🔍 탐색] [📅 기록] [👤 마이]
```
- **배달의민족/요기요 스타일** — 모바일 앱에 가장 익숙한 패턴
- 탭별 독립 콘텐츠 → 홈 스크롤 60% 감소
- 하단 탭바가 footer 광고와 겹칠 수 있어 spacing 조정 필요
- 데스크톱에서 하단 탭이 어색 → 데스크톱은 상단 GNB로 전환 필요

### B. Floating Action + Bottom Sheet (`B_FloatingDrawer.tsx`)
```
[홈: 검색+결과+접이식 카드] + [FAB → 바텀시트(즐겨찾기/캘린더/취향)]
```
- **최소 변경으로 최대 효과** — 라우팅 변경 없음
- 셰프/K-pop/배틀은 `<details>` 접이식으로 처리 → 필요 시만 펼침
- 바텀 시트 UX가 카카오맵/네이버 앱 스타일로 친숙
- 캘린더/즐겨찾기가 한 번 더 깊어지는 단점

### C. Hub & Spoke (`C_HubAndSpoke.tsx`)
```
/[lang]         → 검색+결과만 (매우 가벼움)
/[lang]/explore → 셰프+K-pop+배틀
/[lang]/my      → 캘린더+즐겨찾기+취향
```
- **장기적으로 가장 건강한 구조** — 각 페이지가 독립 번들
- HomeClient 721줄 → ~200줄로 극적 축소
- LCP 최대 개선 (첫 페이지 로드량 최소화)
- /explore, /my 페이지가 개별 SEO 가능
- 구현 비용이 가장 큼 (라우트 3개 추가, 상태 공유 설계)

### D. Sticky Scroll Spy (`D_StickyScrollSpy.tsx`)
```
[Header] → [Search] → [Sticky 섹션 네비] → [인기|대결|즐겨찾기|셰프|K-pop|캘린더]
```
- **코드 변경 최소** — IntersectionObserver + 스크롤 점프만 추가
- 현재 구조 99% 유지, 네비게이션만 추가
- 사용자가 어디에 어떤 기능이 있는지 항상 인지 가능
- 스크롤 길이는 줄지 않음 (탐색성만 개선)

### E. Dashboard Grid (`E_DashboardGrid.tsx`)
```
┌──────────────┬──────────┐
│ 검색+결과     │ 즐겨찾기  │ ← 데스크톱: 2열 그리드
│              │ 캘린더    │
│              │ 취향     │
├──────────────┴──────────┤
│ 셰프 | K-pop             │
└─────────────────────────┘
```
- **데스크톱 최적** — 넓은 화면을 100% 활용
- "내 식단 대시보드" 포지셔닝 → 매일 방문하는 허브
- 사이드바 위젯이 컴팩트하게 모든 정보 표시
- 모바일에서는 1열로 폴백 → 기존과 유사
- 반응형 CSS 복잡도 높음

## 추천 조합

### 모바일 우선 (트래픽 80%+ 모바일): **A 또는 B**
- 빠른 구현: **B** (FAB+Drawer) — 1~2일 구현
- 완성도: **A** (Bottom Tab) — 3~4일 구현

### SEO+성능 우선: **C** (Hub&Spoke)
- 장기적 확장성 최고, 번들 최적화 극대
- 구현 비용 높지만 유지보수 용이

### 최소 변경: **D** (Scroll Spy)
- 반나절 구현 가능, 리스크 최소
- 스크롤 문제 근본 해결은 아님

### 데스크톱 비중 높을 때: **E** (Dashboard)
- 데스크톱 30%+ 트래픽 시 ROI 있음
- "식단 관리" 포지셔닝에 적합
